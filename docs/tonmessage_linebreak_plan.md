# TonMessage 改行制御改善計画

## 背景

- 現状の `TonMessage` は文字を1文字ずつ配置しているだけで、ウィンドウ幅に応じた自動改行を行っていない。
- そのため、長文は右端を超えて描画される可能性がある。
- また、一般的な改行制約である「語の途中で折り返さない」「句読点や閉じ括弧を行頭に出しにくくする」といった制御も未対応である。

## 目的

- `TonMessage` にウィンドウ幅ベースの自動改行を追加する。
- 改行制約の有無を利用側で切り替えられるようにする。
- 既存の手動改行、タグ、ページ送りの挙動は極力維持する。

## 変更対象

- `mononotonka/TonMessage.cs`

## 変更しないこと

- スクリプト記法の大幅な変更はしない。
- 既存の `[next]`、`[blankline]`、`[icon:*]`、`[wait:*]` などの仕様は変えない。
- `TonMessage.cs` 以外へ処理本体を分散させない。

## 追加する公開 API 方針

### 方針

- 言語名ベースのモード名にはしない。
- 「英語用」「日本語用」ではなく、「どの粒度で折り返し候補を扱うか」を名前で表現する。
- 既存 API の `SetTextStyle` などと並んだときに意味が通る命名にする。

### 候補

1. `SetLineBreakMode(MessageLineBreakMode mode)`
   - 意味が最も素直で、今回の改修意図に合う。
2. `SetWrapMode(MessageWrapMode mode)`
   - 短いが、「禁則」まで含むかが少し伝わりにくい。
3. `SetLineBreakRuleMode(MessageLineBreakRuleMode mode)`
   - 意味は明確だが少し長い。

### 推奨案

- メソッド名: `SetLineBreakMode`
- 列挙型名: `MessageLineBreakMode`

### モード案

1. `Character`
   - 文字単位で折り返す。
   - 現状に最も近い単純動作。
   - 自動改行はするが、語途中回避や句読点回避は行わない。
2. `TokenAware`
   - 連続した英数字記号列をなるべくひとまとまりとして扱う。
   - 行頭禁則対象の句読点や閉じ括弧は、可能な限り前行へ残す。
   - 今回ユーザーが求める一般的な改行制約はこちらで対応する。
   - ただし URL やバージョン文字列のような特殊なトークン解析は対象外とする。

## 実装方針

### 1. 自動改行の基本追加

- 文字またはまとまりを配置する前に、次の描画幅を計測する。
- 右端を超える場合は `NewLine` 相当の内部改行を先に行う。
- ただし行頭配置の1要素目が単体で幅を超える場合は、その要素は強制配置する。
  - 無限改行や表示不能を防ぐため。

### 2. 折り返し単位の扱い

- `Character` モード
  - 1文字ずつ配置する。
- `TokenAware` モード
  - 次の単位を以下のようにまとめる。
  - 英数字の連続
  - 英数字に付随する一部記号の連続
  - それ以外の文字は基本1文字単位
  - URL、メールアドレス、バージョン文字列などの特殊なまとまりは今回は考慮しない。
  - ただし、ひとまとまり全体が1行に収まらない場合は語単位保護を諦め、その行に入るところまで通常どおり描画する。
  - 語単位保護は語頭でだけ適用し、長い語の途中では文字単位折り返しへ戻さない。

### 3. 行頭禁則の基本対応

- `TokenAware` モード時のみ有効にする。
- 少なくとも以下を行頭に出しにくくする。
  - `、。`
  - `，．`
  - `,.;:!?`
  - `）】』」〕〉》]})`
  - `"'`
  - `！？`
- 全角文字は句読点系のみを対象とし、小書きかななどの文字そのものは対象外とする。
- 対応方法は「対象文字を単独で次行へ送らない」ことを優先する。
- 直前のまとまりと一緒に前行へ残せない場合は、例外として行頭配置を許容する。
  - 表示不能や無限再試行を避けるため。

### 4. 既存タグとの整合

- 明示改行は従来どおり `NewLine` で処理する。
- `[blankline]` はそのまま活かす。
- `[icon:*]` は1つの描画単位として幅計測し、自動改行対象に含める。
- 文字送り速度は、既存どおり1コマンド消費ベースを維持する。
  - ただし内部で複数文字をまとめて扱う際、見た目の送り方が急変しないよう注意する。

### 5. タグまたぎの扱い

- `TokenAware` モードでは、表示上連続して見える文字列をタグで分断しない。
- 少なくとも以下のような見た目に影響しない制御タグは、語や句読点のまとまり判定を継続できるようにする。
  - `color`
  - `shake`
  - `rotate`
  - `reset`
  - `speed`
- ただし、幅や表示物そのものに影響する要素は別扱いにする。
  - `size`
  - `font`
  - `icon`
- `[var:*]` は展開後の文字列として扱い、通常テキストと同じ判定対象に含める。

### 6. 幅計測の考え方

- `TokenAware` モードの先読みでは、単なる文字列連結ではなく「将来描画される要素列」を基準に幅を見積もる。
- 現在スタイルだけで後続文字をまとめて測ると、`size` や `font` をまたいだケースで実幅とズレるため避ける。
- 先読み中は、必要最小限の仮想スタイル状態を進めながら幅を計測する。

### 7. スペースの扱い

- 折り返し判定では、語間の単一半角スペースを区切りとして扱う。
- 折り返し位置が単一半角スペースに掛かる場合は、次行先頭に残さず吸収する方針とする。
- 連続する半角スペースは意図的なレイアウトの可能性が高いため、原則として保持する。
- 全角スペースは日本語文で意味を持つ可能性が高いため、単一半角スペースと同じ自動吸収は行わない。
- 以上により、「通常の単語間空白は自然に折り返し、意図的な連続空白は壊しにくい」状態を目指す。

### 8. 内部構造の変更方針

- 既存の `MessageCommand` と `DrawObject` は可能な限り活かす。
- 追加候補
  - 改行モード保持用のフィールド
  - 次の描画単位と先読み要素列を解決するための補助メソッド
  - 禁則対象判定用の補助メソッド
  - 先読み中の仮想スタイル状態を扱う補助構造
- 公開 API は最小限に留める。

## 実装手順

1. 改行モード用の公開 API と内部状態を追加する。
2. `Text` と `Icon` の配置直前に自動改行判定を差し込む。
3. `TokenAware` 用の先読み処理を、タグまたぎと仮想スタイル追跡を考慮した形に組み直す。
4. 行頭禁則対象の判定を追加する。
5. 単一半角スペースの吸収ルールと、連続スペース保持ルールを追加する。
6. 既存の明示改行・ページ送りと競合しないことを確認する。

## 確認項目

- 長い日本語文がウィンドウ右端を超えずに折り返されること。
- `TokenAware` モードで、英数字列の途中改行が減ること。
- `TokenAware` モードで、句読点や閉じ括弧が行頭に出にくくなること。
- `TokenAware` モードで、装飾タグをまたいでも語や句読点のまとまり判定が崩れないこと。
- `TokenAware` モードで、単一半角スペースが次行先頭に残りにくいこと。
- 複数半角スペースや全角スペースを含むケースで、意図した空白が不必要に削除されないこと。
- `Character` モードで、単純な文字単位折り返しとして動作すること。
- 既存のサンプルスクリプトで `[next]`、`[blankline]`、`[icon:*]` の動作が崩れないこと。

## 想定リスク

- 文字送りが1文字単位前提のため、描画単位のまとめ方によって見た目のテンポが変わる可能性がある。
- 使用フォントによって記号幅が異なり、禁則の見え方が完全一致しない可能性がある。
- 既存スクリプトが手動改行前提で詰められている場合、自動改行追加で見た目が変わる可能性がある。
- タグまたぎと仮想スタイル追跡を入れることで、先読み処理が複雑化し不具合混入余地が増える可能性がある。
- スペース吸収ルールが強すぎると、意図した整形を壊す可能性があるため、単一半角スペースのみに限定して慎重に扱う。

## 備考

- 実装時は、デフォルト値をどちらのモードにするかを要確認とする。
- 現在実装では `TokenAware` を既定値としている。
- 既存表示との差分確認が必要な場合は、利用側で `Character` を明示設定して比較できる。
